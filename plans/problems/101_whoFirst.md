# 开发路径决策：前端驱动 vs. 后端先行

在构建包含 **CWFrame** 框架及“点亮节点”功能的系统时，开发顺序直接影响到工程的复杂度和迭代效率。

---

## 核心矛盾
项目的核心功能（采集、点亮）本质上是前端的状态展示，即便没有后端也能完成交互；但为了实现**用户信息持久化**（用户名、点亮状态等），必须引入后端支撑。

## 推荐方案：接口先行 (API-First) + 前端 Mock 驱动

**建议路径：** 采用“自顶向下”的视觉反馈，配合“自底向上”的数据契约。

### 1. 为什么不推荐“死磕后端”？
* **反馈周期过长：** 若先花两周配置服务器、数据库和登录逻辑，长时间看不到“点亮”的视觉反馈，容易丧失开发动力。
* **需求不确定性：** 在 UI 交互未定型前，很难精确定义数据库字段。若后期想增加“点亮路径顺序”或“点亮时间”，频繁改动数据库表结构会产生额外内耗。

### 2. 为什么不推荐“纯前端开发”？
* **重构压力巨大：** 若前期前端逻辑写得太随意，后期接入真实异步 API 时，会发现状态管理（State Management）混乱，难以解耦。
* **架构深度不足：** 在工程实践中，如果后端仅作为简单的 CRUD（增删改查）挂件，无法体现出前后端协作的架构思考。

---

## 最佳实践：分阶段迭代流程

### 第一阶段：定义 Data Schema (契约)
在动手编码前，先明确前端界面和后端存储共用的**数据结构**。
> **示例：** 一个“节点”对象应包含哪些属性？`id`, `status`, `userId`, `updatedAt` 等。

### 第二阶段：CWFrame 落地与 Mock 驱动
* **搭建框架：** 优先确定 **CWFrame** 的整体布局与组件化方案。
* **使用 Mock 数据：** 模拟后端 API 返回。这能让你立即看到“点亮”效果，验证交互的合理性。
* **工程化处理：** 在编写前端请求层时，统一使用 **TS** 定义接口，确保后期只需更换 Base URL 即可无缝对接真实后端。

### 第三阶段：后端持久化建设
* 在前端交互逻辑跑通后，开始建设数据库与 API 逻辑。
* 由于前期已定义好数据契约，此时后端开发目标极其明确，能够快速实现逻辑闭环。

---

> **架构提示：** > 建议在前后端交互中全程使用 **TypeScript**。通过共享类型定义（Interfaces），可以确保前端发送的数据与后端接收的结构完全一致，在编译阶段就能规避大部分逻辑错误。
