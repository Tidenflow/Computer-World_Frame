# 参与贡献指南

感谢你关注 `Computer World Frame`。

这个项目目前还处在很早期，但方向已经比较明确了。它不只是一个前端页面项目，而是在逐步变成一个围绕“知识地图、图谱渲染、输入理解”的框架。

如果你愿意参与贡献，非常欢迎。

---

## 项目正在尝试做什么

`Computer World Frame` 想解决的问题不是“如何画一张图”，而是：

- 用户输入一些零散词汇
- 系统理解这些词汇大概指向什么
- 把它们映射到一张结构化知识地图上
- 让这张地图以一种可读、可探索的方式逐步点亮

所以这个项目的核心价值，会逐渐落在下面三层：

- 地图内容本身
- 节点计算与图谱渲染逻辑
- 用户输入的匹配与理解能力

---

## 我们最希望社区帮忙解决的三个难题

如果你只看这一份文档中的一个部分，我最希望你看这里。

因为从长期来看，这个项目最重要、也最值得持续贡献的，就是下面这三个方向。

### 1. Map 生态

我希望这个项目在开源之后，不只是我一个人维护一张默认地图，而是大家都能提供自己的地图。

例如：

- 计算机基础地图
- 前端工程地图
- 后端与系统设计地图
- AI / LLM 地图
- 数据库地图
- 甚至未来扩展到非计算机领域的知识地图

这一方向最需要帮助的地方包括：

- 更好的 map JSON 结构设计
- 更清晰的地图编写规范
- map 校验工具
- map 导入 / 转换工具
- 更容易维护的大地图组织方式
- 如何写出质量更高的节点、依赖和分类

适合这一方向的贡献包括：

- map schema 改进
- map 校验脚本
- map 示例
- map 作者文档
- JSON -> 数据库的转换工具

### 2. 节点计算与图谱渲染逻辑

这已经是当前项目里最复杂的部分之一了。

现在这个项目已经不是在渲染一张静态图，而是在处理很多动态规则，例如：

- Hidden / Outlined / Unlocked 的可见性规则
- 最近一次点亮节点的强调方式
- 图谱聚焦逻辑
- 缩放逻辑
- 树状投影渲染
- 重复节点实例
- 避免共享节点导致整段子树重复爆炸
- 大图情况下如何继续保持可读性

这一方向最需要帮助的地方包括：

- 更好的布局算法
- 更清晰的边路由方式
- 重复节点的视觉策略
- 聚焦和相机策略
- 大图渲染性能
- 复杂节点关系下的可读性优化

适合这一方向的贡献包括：

- 图谱布局算法优化
- 连线视觉优化
- 重复节点策略优化
- 聚焦逻辑优化
- 大图压力测试
- 图形性能优化

### 3. 匹配算法与用户输入理解

从长远来看，这会是这个项目最重要的智能层。

理想状态下，用户输入的并不一定是标准术语，但系统仍然应该尽量理解他的意思。

例如：

- 用户输入一个口语词
- 系统推测它最接近哪个节点
- 或者返回若干可能候选
- 尽量避免误点亮
- 尽量把“用户的真实思路”映射到图上

这一方向未来可能包括：

- 关键词匹配
- 同义词系统
- 模糊匹配
- 权重评分
- embedding 匹配
- 页端 / 本地 LLM 推理
- 轻量本地语义理解

这一方向最需要帮助的地方包括：

- 匹配算法设计
- 评估数据集
- 节点别名系统
- 歧义处理
- 轻量本地推理方案
- LLM 辅助匹配实验

适合这一方向的贡献包括：

- 匹配算法原型
- alias / synonym 支持
- 评分和排序策略
- benchmark 数据集
- 端侧 / 本地推理实验

## 开始之前，希望你先知道这三点

### 1. 不要过度设计

这个项目目前非常强调“先做清晰、可迭代的方案”。

如果你的方案很强，但会显著提高复杂度，请在 PR 描述里说明：

- 它解决了什么问题
- 为什么当前方案不够
- 复杂度增加是否值得

### 2. 尊重当前产品方向

这个项目现在不是在做一个纯学术化的图可视化工具。

当前更重视的是：

- 可读性
- 探索感
- 逐步点亮
- 认知映射
- 底层是图，但屏幕上要可理解

所以一个“数学上更纯粹”的方案，如果让用户更看不懂，未必是更好的方案。

### 3. 关注用户体验，不只关注正确性

这里很多逻辑并不只是“算对”就够了。

还需要思考：

- 用户能不能看懂当前图谱
- 用户能不能理解为什么这个节点被点亮
- 最近一次点亮是不是容易定位
- 黑场探索感有没有被破坏

## Pull Request 建议

请尽量让一个 PR 聚焦一个明确问题。

比较理想的 PR 应该是：

- 一个清楚的问题
- 一个清楚的解决方案
- 如果改了行为，最好有测试
- 能解释为什么这样改更好

尽量避免：

- 把很多不相关的修改塞进同一个 PR
- 大规模重构但没有解释动机
- 一边改算法一边顺手改一堆无关 UI
- 没有测试也没有设计说明的大改动

如果你的 PR 涉及算法或架构，建议写清楚：

- 你观察到了什么问题
- 当前行为为什么不好
- 你改了什么规则或启发式
- 这样改带来了什么 tradeoff

## 什么样的贡献最有价值

一个高质量贡献，通常至少会做到下面的一件事：

- 让图谱更容易读
- 让 map 更容易写和维护
- 让用户输入更容易正确映射到节点
- 在不显著增加复杂度的前提下减少困惑
- 让项目未来的架构路线更清楚
